# claudecode/

This directory manages the `~/.claude` configuration for Claude Code. **Edit files here,
not in `~/.claude` directly** — `install.sh` creates symlinks from `~/.claude` into this
directory, so changes are tracked in version control.

## install.sh

Runs `ln -s` for each artifact into `~/.claude`:

| Source | Destination | Purpose |
|--------|------------|---------|
| `CLAUDE.md` | `~/.claude/CLAUDE.md` | Individual instruction layer loaded into every session |
| `settings.json` | `~/.claude/settings.json` | Hook wiring and Claude Code settings |
| `hooks/` | `~/.claude/hooks/` | Python scripts that run on lifecycle events |
| `commands/` | `~/.claude/commands/` | Slash-command definitions (`/audit`, `/bdd`, etc.) |
| `skills/` | `~/.claude/skills/` | Structured skill bundles for complex workflows |

Re-run `install.sh` after adding new hooks, commands, or skills to activate them.

The script also optionally wires a `knowledge_index` MCP server if the
`CLAUDE_KNOWLEDGE_INDEX_BIN` environment variable points to a binary that accepts
`serve stdio` — useful for semantic search over session logs.

## CLAUDE.md

The **individual instruction layer** of a 3-tier system (individual → org → repo).

- Loaded automatically into every Claude Code session.
- Defines work-level detection (Patch / Task / Feature), code philosophy, anti-patterns,
  architectural preferences, commit discipline, and workflow rules (BDD, TDD, debugging).
- Org and repo `CLAUDE.md` files override specifics; this file sets the baseline.

## settings.json

Wires Python hook scripts to Claude Code lifecycle events:

| Event | Hook(s) | What it does |
|-------|---------|--------------|
| `SessionStart` | `session_summarize.py start` | Records session start time and project |
| `UserPromptSubmit` | `prompt_timestamp.py` | Injects current UTC + local time into context |
| `UserPromptSubmit` | `prompt_questions.py` | Reminds Claude to research before asking questions |
| `UserPromptSubmit` | `prompt_pr_scope.py` | Checks PR metrics; warns if commits/LOC/files exceed thresholds |
| `PreToolUse` (Edit/Write) | `pre_tool_config_guard.py` | Requires approval before editing protected config files |
| `PreToolUse` (Edit/Write) | `pre_tool_spec_guard.py` | Requires approval before editing generated spec test files |
| `PreToolUse` (Edit/Write) | `pre_tool_quality.py` | Blocks edits when uncommitted LOC exceeds 400 lines |
| `PostToolUse` (Edit/Write) | `post_tool_lint.py` | Auto-lints edited files using the appropriate linter |
| `PostToolUse` (Edit/Write) | `post_tool_bypass_warn.py` | Warns if edited file contains lint suppressions or skipped tests |
| `PostToolUse` (Edit/Write) | `post_tool_quality.py` | Tracks cumulative LOC since last commit; writes `quality-state.json` |
| `Stop` | `stop_quality.py` | Soft quality reminder at the end of a response |
| `SessionEnd` | `session_summarize.py end` | Generates structured session summary; persists to `session-logs/` |

The `matcher` field on PreToolUse and PostToolUse entries filters which tools trigger
each hook (e.g., `"Edit|Write|MultiEdit|NotebookEdit"`).

## hooks/

All hooks are self-contained Python scripts with inline `uv` dependency declarations:

```python
#!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.12"
# dependencies = []
# ///
```

This means each hook runs in an isolated environment via `uv run` — no manual venv
setup, no version conflicts, no global pip installs required.

### Hook catalogue

| Script | Event | Purpose |
|--------|-------|---------|
| `pre_tool_config_guard.py` | PreToolUse | Blocks (asks) before editing `go.mod`, `pyproject.toml`, CI configs, Dockerfiles, and other project-wide config files |
| `pre_tool_spec_guard.py` | PreToolUse | Blocks (asks) before editing spec test files (generated by `/codify-spec`, not hand-edited) |
| `pre_tool_quality.py` | PreToolUse | Hard blocks edits when `locSinceCommit` ≥ 400; enforces atomic commit discipline |
| `post_tool_lint.py` | PostToolUse | Detects file extension, runs the first available linter (`ruff`, `golangci-lint`, `shellcheck`, `markdownlint-cli2`, etc.) |
| `post_tool_bypass_warn.py` | PostToolUse | Scans edited files for `//nolint`, `# noqa`, `t.Skip()`, `@ts-ignore`, and similar bypass patterns; warns if found |
| `post_tool_quality.py` | PostToolUse | Runs `git diff --stat HEAD`, sums insertions + deletions, writes count to `~/.claude/quality-state-<hash>.json` |
| `prompt_pr_scope.py` | UserPromptSubmit | Computes commits/LOC/files vs base branch; injects a warning when thresholds are crossed (suppresses for 5 prompts after first trigger) |
| `prompt_questions.py` | UserPromptSubmit | Injects a one-line reminder to research before asking questions |
| `prompt_timestamp.py` | UserPromptSubmit | Injects UTC and local time so Claude always has accurate temporal context |
| `session_summarize.py` | SessionStart / SessionEnd | Logs session start; on end, reads transcript and generates a structured summary in `~/.claude/session-logs/` |
| `stop_quality.py` | Stop | Prints a 4-point quality checklist (methodology, tests, dead code, commit messages) when `stop_hook_active` is true |

Quality state files are **worktree-aware**: the path includes a SHA-256 hash of the
git root to avoid collisions when multiple worktrees are open simultaneously.

### How to add a new hook

1. Create `hooks/<name>.py` with the `uv run --script` shebang.
2. Implement `main()` reading JSON from stdin (Claude passes event data there).
3. Add an entry in `settings.json` under the appropriate event key:

```json
{
  "type": "command",
  "command": "uv run \"$HOME/.claude/hooks/<name>.py\""
}
```

1. Run `install.sh` (the `hooks/` symlink is already in place, so this is usually a no-op,
   but it's good practice).

Exit codes: `0` = success/no-op, `2` = hard block. Output `permissionDecision: "ask"` in
JSON to stdout to require user confirmation before proceeding.

## commands/

Slash-command definitions loaded by Claude Code. Each `.md` file becomes a `/command`
the user (or agent) can invoke. Frontmatter `description` appears in the command picker.

```
commands/
├── audit.md          # /audit — comprehensive code audit
├── bdd.md            # /bdd — force BDD workflow
├── build-spec.md     # /build-spec — interactive spec interview
├── codify-spec.md    # /codify-spec — generate tests from a spec
├── done.md           # /done — completion checklist
├── lint.md           # /lint — run linters for detected project type
├── refactor.md       # /refactor — systematic refactoring
├── review-spec-tests.md  # /review-spec-tests — adversarial spec test review
├── review-spec.md    # /review-spec — stress-test a spec
└── weekly-review.md  # /weekly-review — session log analysis
```

### How to add a new command

1. Create `commands/<name>.md` with a frontmatter `description` and instruction body.
2. Run `install.sh` (symlink already exists; new file is instantly available).
3. Invoke as `/<name>` in Claude Code.

## skills/

Structured skill bundles for multi-phase workflows. Each skill is a **subdirectory**
containing a `SKILL.md` entry point and optional phase files.

```
skills/
├── bdd-orchestrating/    # Full BDD orchestration: intake → scenarios → decompose → implement → done
│   ├── SKILL.md          # Skill metadata and entry instructions
│   ├── DISCOVERY.md      # Intake and context-gathering phase
│   ├── SCENARIOS.md      # Gherkin scenario definition phase
│   ├── DECOMPOSITION.md  # Task breakdown phase
│   ├── TDD.md            # RED → GREEN → REFACTOR implementation phase
│   └── DONE.md           # Completion and cleanup phase
├── debugging/            # Systematic debugging with root cause investigation
│   └── SKILL.md
└── refactoring/          # Small-step refactoring discipline
    └── SKILL.md
```

`SKILL.md` contains a YAML frontmatter block with `name`, `description`, and
`allowed-tools`. Claude Code uses `description` to auto-select the skill based on
the task context.

### How to add a new command/skill

**Simple command** (single `.md`, invoked as a slash command):

1. Create `commands/<name>.md`.
2. Run `install.sh`.

**Structured skill** (multi-file, auto-selected by description):

1. Create `skills/<name>/SKILL.md` with the `name` and `description` frontmatter.
2. Add supporting phase files as needed.
3. Run `install.sh`.

The difference: commands are explicitly invoked (`/name`), skills are auto-selected
by Claude Code based on the task description matching the skill's `description` field.
